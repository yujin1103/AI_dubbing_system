### Slide 6 — Lipsync 통합: LatentSync 1.6 ⭐

**1분 30초**

#### 대본

기술적 핵심 추가입니다. 1차 발표 이후 가장 큰 변화는 **음성과 영상의 입 모양 동기화** 기능입니다.

기존에는 한국어 음성만 영상 위에 덮어씌우는 방식이라, 영어로 말하는 화면 위에 한국어가 나와 입 모양과 발음이 어긋나는 문제가 있었습니다. 이를 해결하기 위해 **LatentSync 1.6**, ByteDance가 공개한 audio-conditioned latent diffusion 모델을 통합했습니다.

처리 과정은 다음과 같습니다. 더빙된 한국어 음성과 원본 영상을 입력으로 받아, 영상의 face crop을 추출한 뒤 **20-step DDIM diffusion**으로 입 영역을 한국어 발음에 맞게 inpaint합니다. 이후 GFPGAN으로 face quality를 향상시킵니다.

LatentSync 1.6은 **512×512 해상도**, **16 프레임 temporal context**로 학습되어 자연스러운 입 움직임이 가능하지만, **한국어 발음에 대한 별도 학습이 필요**한 한계가 있어 별도의 LoRA fine-tuning을 진행했습니다 — 이는 슬라이드 9에서 다루겠습니다.

> 💡 발음 주의: LatentSync (라텐트싱크), DDIM (디디아이엠), inpaint (인페인트), GFPGAN (지에프피갠).

### Slide 7 — 다중 화자 처리: AV-Fusion

**1분**

#### 대본

영상에 여러 사람이 등장하는 경우, **현재 말하는 사람의 얼굴**을 정확히 식별해야 합니다. 이를 위해 세 가지 모델을 결합한 **AV-Fusion** 구조를 적용했습니다.

**DiariZen**이 음성만 보고 화자를 분리하고, **ECAPA**가 각 화자의 음성 지문을 추출합니다. 그리고 **LightASD**는 화면에 보이는 모든 얼굴 중 입 움직임이 음성과 가장 잘 일치하는 얼굴을 찾아냅니다.

이 세 가지가 결합되어, 예를 들어 두 사람이 대화하는 장면에서 한 사람이 말할 때 그 사람의 얼굴만 lipsync가 적용되고, 다른 사람의 얼굴은 원본 그대로 유지됩니다. 화자가 화면에 보이지 않고 목소리만 나오는 경우(off-screen voice)에도 잘못된 얼굴에 lipsync가 적용되지 않도록 처리합니다.

> 💡 "AV-Fusion = Audio-Visual Fusion". 음성 단서(diarization)와 영상 단서(active speaker) 결합으로 단일 모델보다 정확.

### Slide 8 — 후처리: GFPGAN async I/O

**1분**

#### 대본

Lipsync 결과를 그대로 출력하면 face crop 영역이 약간 흐릿할 수 있어, 후처리로 **GFPGAN v1.4**를 적용해 얼굴 영역을 선명하게 복원하고 2배 해상도로 업스케일합니다.

여기서 우리만의 최적화 한 가지를 적용했습니다. **Async I/O 파이프라인**입니다. 기존 GFPGAN 처리 방식은 frame을 한 장씩 읽고, GPU에서 처리하고, 다시 저장하는 순차 방식이었습니다. 이를 **prefetch 4 + write pool 2**의 비동기 파이프라인으로 변경하여, GPU가 frame N을 처리하는 동안 CPU 스레드가 N+1을 미리 읽고, N-1을 동시에 저장하도록 했습니다.

결과적으로 GPU 가동률이 올라가고, 동일 품질에서 **약 22% 시간 단축**을 달성했습니다. 15초 영상 기준 GFPGAN 단계가 4분 30초에서 3분 30초로 줄어듭니다.

> 💡 "동일 품질 + 시간 단축" 명확히 강조. 손실 없이 얻은 최적화.

### Slide 9 — LoRA 다국어 확장

**1분 30초**

#### 대본

LatentSync 1.6은 영어 위주로 학습된 모델이라, **한국어 phoneme**, 즉 ㅂ, ㅍ, ㅁ 같은 양순음에 대한 정확한 입 모양 학습이 부족합니다. 이를 보완하기 위해 한국어 LoRA fine-tuning을 진행했습니다.

데이터셋은 **AI Hub의 한국어 음성-영상 50시간**을 활용했고, RTX 5080 16GB 환경에서 **batch size 2, num_frames 2**로 50,000 step 학습했습니다. 결과 LoRA는 영어 base 모델 대비 한국어 입 모양이 자연스럽게 개선됩니다.

다국어 확장은 이미 **인프라 레벨에서 준비**되어 있습니다. orchestrator가 입력 언어 파라미터에 따라 `latentsync_ko.pt`, `latentsync_ja.pt` 등을 자동으로 선택합니다. 현재 한국어 LoRA가 적용 가능하며, 일본어 데이터 수집이 완료되면 동일 방식으로 학습 가능합니다. 영어와 중국어는 base 모델 자체가 학습되어 있어 별도 LoRA가 필요 없습니다.

> 💡 "이미 학습된 base + 언어별 LoRA 추가" = 4개 언어 지원의 구조적 답.

### Slide 10 — 기술적 도전: sm_120 + nf mismatch ⭐

**1분 30초**

#### 대본

개발 과정에서 가장 큰 두 가지 도전이 있었습니다.

**첫째, RTX 5080의 sm_120 (Blackwell) 호환성 문제**입니다. PyTorch 표준 가속 기법인 DeepCache, channels_last memory format, torch.compile 모두 sm_120에서는 CUDA stream sync 오류 또는 dynamic shape recompile 오류로 인해 사용할 수 없었습니다. SageAttention 2.2.0 같은 최신 attention 가속도 LatentSync UNet의 head_dim 512와 호환되지 않아 폐기했습니다. 결과적으로 **LatentSync 자체의 추론 속도 가속은 현재 sm_120 환경에서 불가능**하다는 결론에 도달했고, 후처리 단계인 GFPGAN async I/O로 가능한 최적화를 진행했습니다.

**둘째, LoRA의 nf mismatch artifact**입니다. 한국어 LoRA는 16GB GPU 메모리 한계로 num_frames 2에서 학습되었지만, 추론은 16에서 진행됩니다. 이 분포 차이로 인해 화자가 측면으로 회전할 때 광대뼈 영역에 약한 색감 자국이 발생합니다. 단기 해결책으로 base 모델을 사용하면 자국은 사라지지만 한국어 정확도가 떨어집니다. 근본 해결책은 **A100 80GB 클라우드 GPU에서 num_frames 16으로 LoRA를 재학습**하는 것입니다. 현재 비용 대비 효율 측면에서 RTX 5080에서 num_frames 4로 점진적 개선을 우선 시도하고 있습니다.

> 💡 핵심 키 메시지: **"불가능한 것은 명확히 검증, 가능한 것은 실제로 적용"**. 모든 시도와 실패를 데이터로 정리.

## Part 2. 새로 추가된 기술 요소 상세 설명

### 2.1 LatentSync 1.6 — Audio-conditioned Latent Diffusion

**모델 구조:**
```
Input video (face crop) → VAE encoder → Latent
                                          ↓
Audio (Whisper embeddings) → Cross-attention
                                          ↓
                              UNet 3D (16 frames)
                                          ↓
                              DDIM Sampler (20 steps)
                                          ↓
                              VAE decoder → Output video
```

**주요 spec:**
- Resolution: 512×512 (face crop)
- Temporal: num_frames=16 (1.6 default)
- Steps: 20 (DDIM)
- VAE: stabilityai/sd-vae-ft-ema
- Trained on: HDTF + VFHQ datasets (영어 위주)

**왜 LatentSync 1.6인가:**
- 1.5는 256 res로 학습되어 입/이 영역 흐림
- 1.6은 512 res + TREPA loss로 high-freq detail 유지
- BD가 공개한 SOTA lipsync 모델 (CVPR 2025)

### 2.2 AV-Fusion (DiariZen + ECAPA + LightASD)

**왜 단일 모델로는 부족한가:**
- Voice-only diarization: 음성만 보면 화자 식별은 되지만 화면 위치 X
- Vision-only ASD: 화면 보면 누가 입 움직이는지 알지만 누가 누구인지 X
- → 음향 + 시각 fusion이 필수

**파이프라인:**
1. **DiariZen**: vocal 추출본을 입력으로 [start, end, speaker_id] 출력
2. **ECAPA**: 각 화자의 voice embedding (512-dim) 추출
3. **LightASD**: 영상 frame별 face_box × audio segment → "speaker score" 계산
4. **Fusion**: 시간대별 active speaker face_box 결정

**LatentSync 적용:**
- Active speaker face_box → LatentSync로 입력
- 다른 face는 원본 유지 → multi-person 영상에서 자연스러움

### 2.3 GFPGAN async I/O 파이프라인

**기존 (sequential):**
```
for f in frames:
    img = imread(f)            # CPU
    out = gfpgan.enhance(img)  # GPU
    imwrite(out)               # CPU
```
GPU가 read/write 동안 idle.

**개선 (async pipeline):**
```
[Reader thread]    -> [Read queue] -> [GPU enhance] -> [Write pool]
                       (prefetch=4)                     (workers=2)
```
GPU가 항상 가동, I/O는 백그라운드. **22% 시간 단축**.

### 2.4 한국어 LoRA — Fine-tuning

**Why LoRA:**
- Full fine-tune은 weight 5GB 저장
- LoRA는 ~100MB delta만 저장 → 다국어 확장 시 합리적

**데이터:**
- AI Hub 한국어 음성-영상 데이터셋 (TS1/10/20/30/40/50)
- 정면 얼굴 위주, 표준어 발음
- 50시간 (필터링 후 사용 가능)

**학습 setup (RTX 5080 16GB):**
- Resolution: 256 (메모리 제약)
- num_frames: 2 (메모리 제약)
- batch_size: 2
- learning_rate: 1e-5
- 50,000 steps, peak VRAM 14.5GB

**알려진 한계:**
- 학습 nf=2, 추론 nf=16 → mismatch
- 광대뼈 영역에 미세한 색감 drift
- 해결: nf=4 점진적 학습 (현재 진행 예정)

### 2.5 sm_120 (Blackwell) 호환성 검증 결과

| 가속 기법 | 결과 | 원인 |
|----------|------|------|
| DeepCache | ❌ Crash | CUDA stream sync issue |
| channels_last | ❌ +65% slower | UNet3D fallback 비효율 |
| torch.compile | ❌ +30% slower | Dynamic shape recompile |
| SageAttention CUDA | ❌ Assertion error | sm_80/sm_89 false negative |
| SageAttention Triton | ❌ Crash | head_dim 512 미지원 |
| 256 resolution | ❌ 입 영역 artifact | LatentSync 1.6은 512 전용 학습 |
| **steps=15** | ⚠️ 한국어 phoneme 정확도 ↓ | 안전 |
| **vae_chunk=4** | ⚠️ 미세 quality drop | 폐기 |
| **GFPGAN async** | ✅ -22% | 채택 |

**결론:** sm_120에서 LatentSync 자체 가속은 사실상 불가능. 후처리 가속만 가능.

## Part 3. 예상 질문 및 답변 대비

### Q1. "LatentSync 자체를 학습한 건가요?"

**A:** 아닙니다. ByteDance가 공개한 사전 학습 모델을 그대로 사용하고, 한국어 phoneme 보강을 위해 **LoRA delta만 별도 학습**했습니다. Full fine-tune이 아닌 LoRA 방식을 택한 이유는 다국어 확장 시 언어별 LoRA만 추가하면 되기 때문입니다.

### Q2. "왜 sm_120 가속이 그렇게 어려운가요?"

**A:** RTX 5080은 2025년 1월 출시된 Blackwell 아키텍처로, PyTorch와 주요 가속 라이브러리들이 아직 완전히 호환되지 않은 상태입니다. CUDA 13 + PyTorch 2.11 nightly까지 사용했으나, DeepCache의 stream sync 가정이나 SageAttention의 head_dim 한계처럼 라이브러리마다 다른 호환성 이슈가 있습니다. 시간이 지나면 자연스럽게 해결될 문제이지만, 현재 시점에서는 클라우드 GPU 사용이 가장 현실적입니다.

### Q3. "광대뼈 미세 자국은 시연에서 보일까요?"

**A:** 정면 발화 시에는 거의 보이지 않고, 측면 회전 시 광대뼈 영역에 약한 색감 시프트가 발생합니다. 시연 영상은 발화 비중이 정면인 클립을 우선 선정했으며, 최종 발표 전까지 num_frames 4 재학습을 통해 50% 추가 감소를 목표하고 있습니다.

### Q4. "왜 base 모델만으로는 안 되나요?"

**A:** Base 모델은 영어 phoneme에 최적화되어 있어 한국어의 ㅃㅍㅁ 같은 양순음이나 받침 발음에서 입 모양이 어색합니다. LoRA로 한국어 데이터를 추가 학습하면 이 정확도가 명확히 개선됩니다. 다만 LoRA 학습 시 num_frames 2로 제약받은 부분이 현재 한계로 남아있고, 이 부분이 광대뼈 자국의 원인입니다.

### Q5. "Docker로 분리한 이유는?"

**A:** 세 가지입니다. 첫째, 환경 격리로 의존성 충돌을 방지합니다 — CosyVoice의 PyTorch 2.0과 LatentSync의 PyTorch 2.11이 한 환경에서 동시 실행 불가능합니다. 둘째, GPU 메모리 분산이 가능합니다 — 향후 멀티 GPU 환경에서 demucs, speaker, tts를 다른 GPU에 배치 가능합니다. 셋째, 재현성이 보장됩니다 — 동일 docker image에서 동일 결과를 보장합니다.

### Q6. "다국어 4개 지원 — 일본어/중국어 결과도 보여줄 수 있나요?"

**A:** 일본어는 LoRA 학습용 데이터 수집 단계에 있어 현재는 base 모델로만 동작합니다. 중국어는 LatentSync 1.5가 중국어 데이터를 추가 학습한 모델이라 base만으로 잘 동작합니다. 영어는 원본 그대로이므로 lipsync가 필요 없는 케이스입니다. 시연에서는 한국어 결과를 중심으로 보여드리고, 다국어 확장 인프라가 준비되어 있음을 강조합니다.

### Q7. "MOS 모델은 이번에도 사용되나요?"

**A:** 1차 발표 때 발표한 MOS 자동 검증 모델은 그대로 사용하고 있습니다. 화자별 voice profile 추출 시 후보 reference 음성을 MOS 점수로 정렬하여 가장 높은 품질의 reference를 자동 선택합니다. 또한 TTS 합성 결과가 MOS 3.5 미만이면 자동 재합성을 트리거합니다. 이번 발표에서는 새로 추가된 기술 중심으로 다루기 위해 MOS는 별도 슬라이드를 두지 않았습니다.

### Q8. "처리 시간이 너무 길지 않나요?"

**A:** 현재 100초 영상에 75-80분이 소요되며, 이는 실시간보다 약 45배 느린 속도입니다. 이는 LatentSync diffusion이 frame당 20-step 추론을 수행해야 하기 때문에 본질적으로 무거운 연산입니다. 사전 처리 후 결과 영상을 시청하는 use case에는 적합하지만, 실시간 스트리밍에는 부적합합니다. 이 부분은 클라우드 GPU 또는 더 가벼운 lipsync 모델 (Wav2Lip 등)로 보완 가능하며, 최종 발표에서 trade-off로 다룰 예정입니다.
